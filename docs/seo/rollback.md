# SEO Rollback Guidance

Safe rollback procedures for the `lib/seo/*` module (SEO Architecture v1.0).
Prefer the smallest, most targeted rollback that solves the problem — env
and per-route config changes over reverting code, and code reverts over
deleting the framework.

## 1. Disable indexing sitewide (fastest, safest, no deploy required if your host supports live env edits)

Two independent levers gate every route's *effective* index/sitemap state
(`lib/seo/indexability.ts`); flipping either one to "off" makes every route
non-indexable and empties the sitemap, regardless of any route's declared
`index: true`:

- Set `SEO_INDEXING_ENABLED=false` (or unset it — it defaults to `false`).
- Or deploy in a way that is not a "recognized production" SEO environment
  (`NODE_ENV !== "production"`, or `NEXT_PUBLIC_APP_URL` missing/invalid).
  This is fail-closed by design: an unrecognized/misconfigured deploy can
  never accidentally index.

After changing either variable, **redeploy/restart** the app — `process.env`
is read at request/build time, so editing your host's env store alone does
not take effect until the process restarts. Confirm with:

```bash
NEXT_PUBLIC_APP_URL=https://<production-origin> npm run seo:check
```

`summary.environment.effectiveIndexingActive` must read `false`, and
`summary.indexedRoutes` / `summary.sitemapRoutes` must both be `0`.

## 2. Per-route rollback (revert a single route to noindex)

Edit the affected route's entry in `lib/seo/routes.ts`:

```ts
index: false,
sitemap: false,
follow: true, // keep true unless you also want to block link discovery
```

This is the correct rollback for "we indexed a tool too early" — it does
not require touching env, metadata, or sitemap/robots code. Redeploy after
the edit; the sitemap/robots routes and root/route metadata all read from
this same registry, so a single edit here is sufficient.

## 3. Metadata rollback (revert a migrated page's title/description/OG)

- Metadata is generated by `lib/seo/metadata.ts#buildMetadataForRoute` /
  `buildRootMetadata` from `lib/seo/routes.ts` route data — there is no
  separate metadata file per page to hand-edit.
- To restore a prior title/description/OG image for one route, edit that
  route's `title` / `description` / `ogImage` fields in `lib/seo/routes.ts`.
- To fully revert the metadata *framework* for a page (e.g. suspected bug in
  the builder itself, not just wrong copy), revert the specific commit that
  migrated that page to centralized SEO (see `git log --oneline -- app/`
  for the migration commits) rather than hand-patching generated output.

## 4. Verification rollback (stop emitting a verification tag)

- Clear the offending variable: set `SEO_GOOGLE_SITE_VERIFICATION=` or
  `SEO_BING_SITE_VERIFICATION=` (blank) and redeploy. A blank/missing token
  is omitted from `Metadata.verification` entirely — no empty tag is ever
  emitted, and this is always non-fatal (`buildRootMetadata` never throws
  for a missing or placeholder-looking token).
- If a **placeholder-looking** token was accidentally configured (e.g.
  `your-token-here`), no runtime rollback is even required for the
  page output — `buildRootMetadata` already omits it automatically
  (`lib/seo/verification.ts#isPlaceholderVerificationToken`). It will,
  however, still fail `npm run seo:check` loudly
  (`seo/placeholder-verification-token`) until the env var is corrected or
  cleared, by design, so CI/audits catch the misconfiguration.
- Removing a *real* verification token unverifies the property in
  GSC/Bing on their side after some time — only do this intentionally (e.g.
  decommissioning a property), and coordinate with whoever manages the
  Search Console / Webmaster Tools account.

## 5. Sitemap / canonical verification after any rollback

After any rollback above, confirm the live surface matches expectations:

```bash
NEXT_PUBLIC_APP_URL=https://<production-origin> npm run seo:check
```

- `/sitemap.xml` should list only the routes you expect to still be
  indexable (compare against `reports/seo-audit.md`'s route table).
- Run Search Console **URL Inspection** on the home page and one other
  previously-indexable route to confirm the live canonical still matches
  `NEXT_PUBLIC_APP_URL` + the route's path, and that "Indexing allowed"
  matches the rolled-back robots directive.
- `robots.txt` (`/robots.txt`) always allows `/` and disallows `/api/`
  regardless of indexing state — a rollback never needs to touch
  `lib/seo/robots.ts` itself; robots blocking `/` is not part of this
  framework's rollback story.

## 6. Deployment implications

- Every lever above (env vars, `routes.ts` flags) takes effect only after a
  **new deploy or process restart** — there is no live/dynamic toggle.
  Budget rollback time accordingly; it is not instantaneous.
- `NEXT_PUBLIC_*` values are baked into the production build artifact at
  build time (see root `AGENTS.md`). Changing `NEXT_PUBLIC_APP_URL` always
  requires a rebuild, not just a redeploy of a previous artifact.
- Prefer config/env rollback (sections 1–4) over deleting or bypassing
  `lib/seo/*` itself — the framework is the single source of truth for
  metadata, JSON-LD, sitemap, and robots. Only revert framework code if the
  *framework itself* is at fault (e.g. a builder bug), not for a
  content/policy mistake, which sections 1–4 already cover without a code
  change.
