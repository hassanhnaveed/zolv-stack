# Search Console / Bing Webmaster Setup (Runbook)

Operational runbook for Google Search Console (GSC) and Bing Webmaster Tools
verification. Binding companion to `docs/seo/url-policy.md` and
`lib/seo/config.ts`. No real domain names, credentials, or tokens belong in
this repository — every value below is illustrative.

## One-time setup (per environment/domain)

1. **Purchase and point the production domain.** Until then, keep
   `NEXT_PUBLIC_APP_URL` set to a placeholder HTTPS origin for builds (see
   `docs/seo/rollback.md` and `.env.example`).
2. **Prefer a Google Search Console Domain property**, not a URL-prefix
   property. A Domain property covers `http`/`https` and all subdomains with
   one verification, which survives future subdomain changes.
   - Domain properties verify via **DNS TXT record** only (no HTML tag
     option). Add the TXT record GSC gives you to your DNS provider, then
     click "Verify" in GSC. This does not require deploying any code.
   - If you must use a URL-prefix property instead (e.g. a shared host where
     you cannot manage DNS), use the **HTML tag** method: GSC gives you a
     token, which you place in `SEO_GOOGLE_SITE_VERIFICATION` (see
     "Env / build timing" below) — never edit `app/layout.tsx` by hand.
3. **Bing Webmaster Tools: import from GSC when possible.** Bing supports
   importing a verified Google Search Console property directly, which
   verifies Bing without a second manual step. Use that path first.
   - If import isn't available, verify Bing via its own HTML meta tag
     method and place that token in `SEO_BING_SITE_VERIFICATION`.
4. **Submit the sitemap** in both consoles once verification succeeds:
   `{origin}/sitemap.xml` (e.g. `https://example.com/sitemap.xml`). The
   sitemap route is generated by `lib/seo/sitemap.ts` — never hand-author or
   hand-submit a different sitemap URL.

## Env / build timing (HTML-tag method)

- `SEO_GOOGLE_SITE_VERIFICATION` and `SEO_BING_SITE_VERIFICATION` are
  **server-only** environment variables (never `NEXT_PUBLIC_*`) read by
  `lib/seo/config.ts#getSeoConfig()`.
- `lib/seo/metadata.ts#buildRootMetadata()` wires whichever tokens are set
  into Next's native `Metadata.verification` field on the root layout only
  (site-wide meta tags, not per-page):
  - Google → the native `verification.google` key → renders
    `<meta name="google-site-verification" content="...">`.
  - Bing → the `verification.other["msvalidate.01"]` key (Next has no native
    Bing field) → renders `<meta name="msvalidate.01" content="...">`.
- In this project's Lightsail artifact flow, root verification metadata and
  sitemap/indexing decisions are evaluated during the GitHub Actions
  production build. Configure both verification tokens as GitHub Environment
  secrets and `SEO_INDEXING_ENABLED` as a GitHub Environment variable for the
  `production` environment used by the build job, then redeploy through CI.
  Setting these values only on the Lightsail host after the artifact was built
  cannot update already-baked static HTML or build-time indexing output.
- A token left blank is omitted entirely — no empty `<meta>` tag is ever
  emitted. Missing verification is never a build failure.
- A token that still looks like an unfilled placeholder (e.g.
  `your-token-here`, `REPLACE_ME`, containing the literal word `example`) is
  **never emitted** to metadata even if configured, and separately fails
  `npm run seo:check` loudly with `seo/placeholder-verification-token` so a
  placeholder cannot silently ship to production. See
  `lib/seo/verification.ts#isPlaceholderVerificationToken`.

## Per-deploy checklist

Run before every production deploy that could affect indexing, canonicals,
or verification:

```bash
NEXT_PUBLIC_APP_URL=https://<production-origin> npm run seo:check
NEXT_PUBLIC_APP_URL=https://<production-origin> npm run build
```

- The local pre-push hook runs `npm run seo:check`; ensure
  `NEXT_PUBLIC_APP_URL` is set to a valid HTTPS origin in the developer
  environment (for example via `.env.local`).
- Confirm `reports/seo-audit.md` shows `0` errors. Warnings (title/description
  length) are advisory and never block a deploy.
- Confirm `summary.environment.origin` in the audit matches the real
  production origin — a mismatch means `NEXT_PUBLIC_APP_URL` is wrong for
  this deploy.
- If `SEO_INDEXING_ENABLED` or a verification token changed this deploy,
  redeploy is required (see "Env / build timing" above) before the change is
  live.

## Per-release checklist (after a production deploy goes live)

1. In GSC, open **URL Inspection** for the home page and one representative
   indexable route (see `lib/seo/routes.ts` for what is currently
   `index: true`). Confirm:
   - The **live URL** matches the canonical Search Console detects.
   - "Indexing allowed" is `Yes` (robots directive matches
     `lib/seo/indexability.ts`'s effective state for this deploy).
2. Request indexing/re-crawl only for pages that materially changed —
   avoid mass "request indexing" spam, which does not speed up crawl budget
   and can look like abuse.
3. In Bing Webmaster Tools, confirm the imported/verified site shows the
   same sitemap and a matching verified status.
4. Re-check verification health any time `SEO_GOOGLE_SITE_VERIFICATION` /
   `SEO_BING_SITE_VERIFICATION` change: `npm run seo:check` prints
   `Verification — google: <configured|missing|placeholder>, bing: ...`.

## Current indexing scope (see `lib/seo/routes.ts` for the live source of truth)

As of this runbook, the non-tool routes with already-approved `index: true` /
`sitemap: true` declarations are Home, the Fileora hub, About, Contact,
Security, Privacy, and Terms. Every Fileora tool route stays `index: false` /
`sitemap: false` (Task 9's index quality gate approved zero tools initially —
see the comment above `TOOL_ROUTE_DEFAULTS` in `lib/seo/routes.ts`). Do not
submit an individual tool URL for indexing in GSC/Bing until its route is
flipped to `index: true` in the registry and redeployed.
